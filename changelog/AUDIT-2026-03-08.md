## 4. Migração MySQL → PostgreSQL (Supabase)

### 4.1 Barreiras identificadas (77 tabelas, ~56 arquivos PHP com SQL)

| Sintaxe MySQL | Equivalente PostgreSQL | Status |
|---|---|---|
| `AUTO_INCREMENT` | `GENERATED ALWAYS AS IDENTITY` | Requer conversão no schema |
| `` `backticks` `` | `"double_quotes"` | Requer conversão em TODAS as queries |
| `TINYINT(1)` | `BOOLEAN` | Requer conversão + cast nos resultados |
| `INT(4)`, `INT(11)` | `INTEGER` / `BIGINT` | Display width ignorado em PostgreSQL |
| `ENUM('a','b')` | `TEXT CHECK (col IN ('a','b'))` | Requer conversão |
| `LONGBLOB` | `BYTEA` | Imagens — requer conversão |
| `DATE_ADD(d, INTERVAL n MONTH)` | `d + INTERVAL 'n months'` | ~20 ocorrências em queries.php |
| `DATE_FORMAT(d, '%Y-%m-%d')` | `TO_CHAR(d, 'YYYY-MM-DD')` | Múltiplas ocorrências |
| `CURDATE()` | `CURRENT_DATE` | Múltiplas ocorrências |
| `NOW()` | `NOW()` | **Compatível** |
| `GROUP_CONCAT()` | `STRING_AGG()` | Múltiplas ocorrências |
| `MATCH() AGAINST()` | `tsvector` / `to_tsquery()` | 1 tabela (ocorrencias.oco_tag) |
| `IF(cond, a, b)` | `CASE WHEN cond THEN a ELSE b END` | Múltiplas ocorrências |
| `ENGINE=InnoDB` | (omitir) | Remover do schema |
| `CHARACTER SET utf8` | `ENCODING 'UTF8'` no CREATE DATABASE | Remover das tabelas |
| `DEFAULT CURRENT_TIMESTAMP` | **Compatível** | OK |
| `ON UPDATE CURRENT_TIMESTAMP` | **Não existe** — usar trigger | Requer trigger em tickets_extended |

### 4.2 Schema PostgreSQL

Arquivo gerado: `install/5.x/09-POSTGRESQL-SUPABASE-SCHEMA.sql`

### 4.3 CoffeeCode DataLayer — Bloqueador Crítico

O ORM `coffeecode/datalayer` é MySQL-específico (usa backticks, funções MySQL, DSN mysql).
**A API REST NÃO funcionará com PostgreSQL sem trocar o DataLayer.**

**Opções:**
1. **Recomendado:** Substituir por `illuminate/database` (Eloquent standalone) — suporta PostgreSQL
2. Alternativa: Substituir por `doctrine/dbal`
3. Alternativa: Migrar para queries PDO raw com PostgreSQL

**Impacto:** Todos os Controllers da API e Models que estendem DataLayer precisarão ser refatorados.
**Estimativa:** 30+ arquivos em `api/ocomon_api/app/`.

**Ação imediata:** Adicionar ao composer.json do API:
```bash
composer require illuminate/database
composer remove coffeecode/datalayer coffeecode/paginator coffeecode/router
```

## 5. Google OAuth via Supabase Auth

### 5.1 Fluxo implementado (PKCE — Authorization Code Flow)

```
Usuário clica "Entrar com Google"
  → login.php redireciona para SupabaseAuth::getAuthorizationUrl()
  → Supabase Auth redireciona para Google
  → Usuário autentica com conta @workspace.com
  → Google redireciona para Supabase
  → Supabase redireciona para /includes/common/oauth_callback.php?code=...
  → oauth_callback.php troca code por token via Supabase API
  → Valida domínio do email (@workspace.com)
  → Cria ou atualiza usuário na tabela `usuarios`
  → Cria sessão PHP com dados do usuário
  → Redireciona para index.php
```

### 5.2 Configurações necessárias no Supabase Dashboard

1. **Authentication > Providers > Google**: habilitar, configurar Client ID e Secret do Google Cloud Console
2. **Authentication > URL Configuration > Redirect URLs**: adicionar `https://seu-dominio.com/includes/common/oauth_callback.php`
3. **Authentication > Email**: opcionalmente restringir domínios (@workspace.com)

### 5.3 Google Cloud Console

1. Criar projeto ou usar existente
2. APIs & Services > Credentials > OAuth 2.0 Client ID
3. Authorized redirect URIs: `https://SEU_PROJECT.supabase.co/auth/v1/callback`
4. Authorized JavaScript origins: `https://seu-dominio.com`
5. Restringir ao Workspace: Google Admin > Security > OAuth

### 5.4 Constantes necessárias em config.inc.php

```php
SUPABASE_URL            // https://xxxxx.supabase.co
SUPABASE_ANON_KEY       // eyJ... (chave pública)
SUPABASE_SERVICE_KEY    // eyJ... (chave privada — NUNCA no frontend)
SUPABASE_JWT_SECRET     // Segredo para validar JWTs
GOOGLE_WORKSPACE_DOMAIN // Ex: "laboratoriosobral.com.br"
OAUTH_CALLBACK_URL      // https://seu-dominio.com/includes/common/oauth_callback.php
```

## 6. Achados de Segurança — Backend

### CRÍTICO: SQL Injection

**Arquivo:** `includes/common/auth_process.php` — linhas 159-174 (INSERT LDAP)
**Arquivo:** `new_user_process.php` — linhas 155-163 (INSERT novo usuário)
**Arquivo:** `auth_process.php` — linha 329 (UPDATE hash — concatenação)

**Causa:** Concatenação direta de variáveis em queries SQL em vez de prepared statements.

**Mitigação imediata:** Com Google OAuth, o fluxo LDAP e de cadastro manual será descontinuado.
Os blocos de código vulneráveis estarão inacessíveis quando AUTH_TYPE=GOOGLE_OAUTH.

**Ação necessária (mesmo assim):** Converter para prepared statements antes de qualquer deploy.

### CRÍTICO: Session Fixation

**Arquivo:** `includes/common/auth_process.php`
**Problema:** Ausência de `session_regenerate_id(true)` após login bem-sucedido.
**Fix:** Adicionado em auth_process.php na seção GOOGLE_OAUTH.

### CRÍTICO: JWT com chave fraca (API)

**Arquivo:** `api/ocomon_api/app/Models/AccessToken.php`
**Problema:** JWT assinado com `$user->data()->password` — chave derivada da senha do usuário.
**Fix (futuro):** Migrar API para validar tokens Supabase JWT (RS256 centralizado).

### ALTO: Hashing MD5 legado

**Arquivo:** `includes/functions/dbFunctions.php` — função `pass()`
**Problema:** Aceita senhas em MD5 plain text para compatibilidade com versões antigas.
**Mitigação:** Com Google OAuth, nenhuma senha será armazenada ou validada localmente.

## 7. Achados de Segurança — Frontend

### CRÍTICO: innerHTML com resposta AJAX

**Arquivo:** `includes/javascript/funcoes-3.0.js` — linhas 843, 922, 929
```javascript
ajaxDisplay.innerHTML = ajaxRequest.responseText; // XSS
```
**Fix:** Substituir por `ajaxDisplay.textContent` onde o conteúdo é texto, ou garantir que o servidor retorne HTML sanitizado.

### ALTO: XSS em echo sem htmlspecialchars

**Arquivo:** `ocomon/geral/ticket_show.php` — linha ~881
```php
echo "<option value='".$worker['user_id']."'>".$worker['nome']."</option>"; // sem escape
```
**Fix:** Usar `htmlspecialchars($worker['nome'], ENT_QUOTES, 'UTF-8')`.

### ALTO: jQuery .html() com dados do servidor

**Arquivo:** `login.js`, `dashboard.php`
```javascript
$('#divResult').html('<div>' + response.message + '</div>'); // XSS se message contiver HTML
```
**Fix:** Usar `$('#divResult').text(response.message)` ou sanitizar server-side garantindo HTML válido.

### MÉDIO: $_SERVER['PHP_SELF'] em action de formulários

**Presente em:** múltiplos arquivos (ticket_add.php, ticket_edit.php, etc.)
```php
<form action="<?= $_SERVER['PHP_SELF']; ?>">
```
**Fix:** Usar `htmlspecialchars($_SERVER['PHP_SELF'], ENT_QUOTES, 'UTF-8')`.

### MÉDIO: Variáveis de sessão sem escape em HTML

**Arquivo:** `index.php` — linhas 173, 182
```php
<span><?= $_SESSION["s_usuario_nome"] ?></span>
```
**Fix:** `<?= htmlspecialchars($_SESSION["s_usuario_nome"], ENT_QUOTES, 'UTF-8') ?>`
